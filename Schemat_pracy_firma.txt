Praca w firmie schemat pracy z gitem


Wchodzę do firmy, robię kawkę, odpalam komputerk i pierwsze co robię:

1. Rozgrzewka (Terminal)
Otwierasz terminal, wchodzisz do folderu projektu i sprawdzasz, gdzie w ogóle jesteś.

git status – sprawdzasz, czy nie zostawiłaś czegoś rozgrzebanego wczoraj (jeśli terminal jest "czysty", idziesz dalej).

git checkout main – zawsze zaczynasz od "bazy".

2. Pobranie "Świata"
To najważniejszy krok. Przez noc inni programiści mogli wrzucić 100 poprawek. Musisz je mieć, żeby nie pracować na starej wersji.

git pull origin main – pobierasz nowości. Teraz Twój komputer wie to samo, co serwer.

3. Planowanie (Nowy Branch)
Nigdy nie psujesz głównego kodu. Chcesz dodać nową funkcję? Tworzysz dla niej osobny "świat".

git checkout -b feature/moje-zadanie – (np. feature/analiza-danych). Od teraz wszystko, co zrobisz, dzieje się tylko na tej gałęzi.

4. Kodowanie (Pętla pracy)
Piszesz kod, sprawdzasz, czy działa. Co 1–2 godziny robisz "zapis gry":

git add .

git commit -m "Dodano wstępny wykres zależności"

5. Przerwa na obiad / Koniec dnia (Backup)
Nawet jeśli zadanie nie jest skończone, warto wysłać je na serwer, żeby szef widział, że praca idzie do przodu, i żebyś miała kopię zapasową.

git push origin feature/moje-zadanie – Twoja praca leci do chmury.

Sytuacja	Co wpisać?
"Zepsułam plik i chcę, żeby był taki jak rano"	:    
git restore nazwa_pliku

"Napisałam głupoty w ostatnim commicie"       :	
git commit --amend -m "Nowy lepszy opis"

"Zrobiłam totalny bałagan i chcę wszystko skasować do zera"	:  
git reset --hard HEAD (Uważaj! To kasuje wszystko od ostatniego zapisu!)

"Gdzie ja w ogóle jestem?"	
git branch (Gwiazdka pokaże Ci Twoją gałąź)


W firmie nikt nie krzyczy za błędy w kodzie, ale ludzie denerwują się, gdy:

Ktoś zapomni zrobić pull i nadpisze czyjąś pracę (tzw. konflikt).

Ktoś robi commity z opisem "asdfg" albo "poprawka1" (nikt nie wie, o co chodzi).

Ktoś pracuje cały tydzień bez robienia push (ryzyko utraty pracy).

GIT MUST HAVE

1.
git stash 
– „Chowaj fanty!”
Wyobraź sobie: pracujesz nad nowym wykresem, masz rozgrzebany kod, pliki nie działają, a nagle szef mówi:
 „Ewelina, musisz natychmiast poprawić błąd na main!”.

Nie chcesz robić commita z niedokończonym kodem.

Wpisujesz: git stash.

Twoje zmiany znikają (są bezpiecznie schowane w „schowku”), a folder jest czysty. 
Możesz przełączyć się na main.

Gdy wrócisz, wpisujesz: 
git stash pop 
i Twoja rozgrzebana praca wraca!

2. 
git diff 
– „Co ja tu właściwie zmieniłam?”
Zanim zrobisz git add ., zawsze warto sprawdzić, czy nie zostawiłaś w kodzie jakiegoś bałaganu albo 
komentarza „tu jest błąd, naprawić później”.

git diff – pokazuje zmiany, których jeszcze nie dodałaś do poczekalni.

git diff --staged – pokazuje, co już dodałaś i co zaraz poleci w commicie.

3. 
git commit --amend 
– „Ups, literówka!”
Zrobiłaś commita, ale zapomniałaś dodać jednego pliku albo w opisie zrobiłaś błąd? 
Nie rób nowego commita „Poprawka poprawki”.

Wpisz: git commit --amend -m "Poprawny opis".

To nadpisze Twój ostatni, błędny commit tym nowym, czystym.


4. 
git log --oneline --graph 
– „Mapa bitwy”
Zwykły git log jest nudny i długi. Jako specjaliści lubimy widzieć drzewo projektu:

git log --oneline --graph --all 
– pokaże Ci piękną mapę branchy, kropek i kresek. 
Od razu widzisz, gdzie się odgałęziłaś i gdzie był merge.

5.
git fetch 
– „Co tam u innych?”
Wspominałaś o git pull. Pull robi dwie rzeczy: pobiera dane i od razu je łączy z Twoimi. Czasami to ryzykowne.

git fetch – mówi: „Sprawdź, co się zmieniło na serwerze, ale nic nie ruszaj u mnie”. Dzięki temu widzisz zmiany, 
ale Twój kod pozostaje nietknięty, dopóki sama nie zdecydujesz o merge.

6. 
Wchodzisz, kawa paruje. Zamiast rzucać się na kod, robisz tzw. Sanity Check:

git remote -v – sprawdzasz, czy na pewno jesteś podpięta pod dobre repozytorium
 (często w firmach mamy ich kilka).

git branch -a 
– patrzysz, jakie branche są na serwerze. Może kolega właśnie wrzucił coś ważnego?

Conflict Management – to najważniejsza umiejętność. Gdy robisz merge i Git mówi „CONFLICT”, nie panikuj. 
To nie błąd, to prośba Gita: „Ewelina, Ty i Janek zmieniliście tę samą linię. 
Wybierz, która wersja jest lepsza”.
Otwierasz plik, kasujesz znaczniki <<<< i >>>>, zostawiasz dobry kod i robisz add oraz commit.


CO JESZCZE LIZNĄĆ?
1.
. git rebase 
– "Sprzątanie historii"
To jest temat, który dzieli programistów na dwa obozy.

O co chodzi? Merge tworzy w historii "pajączka" (dodatkowy commit łączący). 
Rebase przesuwa Twoje zmiany na samą górę aktualnego maina, tak jakbyś dopiero co zaczęła nad nimi pracować.

Dlaczego warto? Historia projektu wygląda jak prosta linia, a nie poplątane kable. 
To jest bardzo cenione w dużych projektach.

Zasada Seniora: Nigdy nie rób rebase na branchach, na których pracują też inni ludzie!

2. 
git cherry-pick 
– "Wyciąganie rodzynek"
Wyobraź sobie, że kolega na swoim branchu naprawił błąd w pliku, którego Ty też używasz. 
Nie chcesz całego jego brancha (bo ma tam 100 innych rzeczy), chcesz tylko ten jeden konkretny commit.

Komenda: git cherry-pick <numer_commita>.

Git bierze tylko ten jeden "puzzel" i dokleja go do Twojej pracy. To ratuje życie w sytuacjach kryzysowych.

3. .gitignore 
– "Sztuka ignorowania"
W Twoim git status widziałem pliki .ipynb. W Data Science to standard, ale Git nie lubi dużych zbiorów danych 
(np. CSV o wielkości 1GB).

Musisz wiedzieć, jak skonfigurować plik .gitignore, żeby Git nie próbował wysyłać na serwer ciężkich danych, 
haseł do bazy (plików .env) czy śmieci systemowych.

Pro-tip: Na stronie gitignore.io wpisujesz "Python" lub "Jupyter" i dostajesz gotowca.

4. Rozwiązywanie konfliktów (Conflict Resolution)
To nie jest komenda, to umiejętność. W biurze konflikty to codzienność. Musisz umieć czytać znaczniki:

<<<<<<< HEAD (Twoje zmiany)

======= (Granica)

>>>>>>> branch-kolegi (Zmiany kogoś innego)
Twoim zadaniem jest usunąć te linie i zostawić czysty kod. Seniorzy robią to spokojnie, juniorzy panikują. Bądź jak senior.

5. Git Hooks 
– "Automatyczny strażnik"
To są skrypty, które uruchamiają się same przed commitem.

W firmach często używa się ich, żeby np. automatycznie sprawdzić, 
czy nie zostawiłaś w kodzie błędów składniowych albo czy nie zapomniałaś o dokumentacji. 
Warto wiedzieć, że coś takiego istnieje, żebyś nie zdziwiła się, 
gdy git commit nagle "odmówi posłuszeństwa", bo kod jest niechlujny.

6. Praca z Notebookami (nbdime)
Jako Data Scientist będziesz pracować w Jupyter Notebooks. 
Git średnio radzi sobie z porównywaniem plików .ipynb (pokazuje dziwne kody JSON zamiast zmian w komórkach).

Warto zainstalować narzędzie nbdime. 
Pozwala ono robić "ludzkie" diffy i merge notebooków bezpośrednio w przeglądarce. 
To absolutny game changer w Data Science.
